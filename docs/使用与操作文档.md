# Ivyea Agent 使用与操作文档

> 自托管的**亚马逊运营对话式 Agent**。专长广告巡检（确定性规则引擎 + LLM 复核 + 护栏 + 审核制写入 + 审计回滚），也能干日常运营杂活（读 Excel、算数、写报告、查网页）。
> 哲学：**证据驱动、写操作一律人工审批、数据私有**。

---

## 一、安装

```bash
# 方式 A：装 Release 里的 wheel（任何人可用）
pip install https://github.com/Hector-xue/ivyea-agent/releases/download/v0.4.0/ivyea_agent-0.4.0-py3-none-any.whl

# 方式 B：从源码（pipx，装完任意目录敲 ivyea）
pipx install git+https://github.com/Hector-xue/ivyea-agent.git

# 方式 C：克隆后开发安装
git clone https://github.com/Hector-xue/ivyea-agent.git && cd ivyea-agent
pip install -e ".[anthropic]"     # anthropic 可选(用 Claude 才需要)
```
跨平台纯 Python（≥3.9），Windows/macOS/Linux 均测过。

## 二、首次运行（引导）

直接敲 `ivyea`，**首次会自动进引导**（也可手动 `ivyea onboard`）：

1. **选主脑模型并配 key/provider** —— 推荐 DeepSeek（便宜、工具调用稳）；也可 Claude API（需 `ANTHROPIC_API_KEY`）、Gemini API（需 `GEMINI_API_KEY`）、AWS Bedrock（需 `boto3` + AWS 凭据链）、OpenRouter、Kimi、通义、Z.AI、MiniMax、xAI、Ollama、本地/自定义 OpenAI 兼容端点。
2. **（可选）接领星 OpenAPI** —— 拉真实广告数据 / 写回。填 host / appId / appSecret。
3. **（可选）生成 `AGENTS.md`** —— 写你的长期打法/边界，对话时自动注入。

配置都存在本机 `~/.ivyea/`（`.env` 密钥 600 权限，`settings.json`，`mcp.json`）。

## 三、两种用法

### 1) 对话模式（推荐）
```bash
ivyea                      # 进对话；裸敲即可
ivyea --continue           # 续接上次会话
ivyea chat --resume <id>   # 续接指定会话
ivyea --raw                # 原始流式输出（默认 Markdown 渲染）
```
直接说人话，它自己编排工具：「看下 sid 1863 这周广告」「读 /path/report.csv 帮我分析」。写操作一律弹**人工审批**。

**斜杠命令**：
| 命令 | 作用 |
|---|---|
| `/help` | 命令与用法 |
| `/model` `/config` | 切模型/配 key、配置向导 |
| `/plan` `/approve` | 进/出**计划模式**（只读，不写）；批准后执行 |
| `/cost` | 本会话 + 今日 token 用量与成本估算 |
| `/compact` | 手动压缩上下文（LLM 摘要历史，省 token，摘要入库） |
| `/compact auto on/off/status` | 开启、关闭或查看自动压缩；默认关闭，长任务只提醒不打断 |
| `/init` | 生成账户指令模板 `AGENTS.md` |
| `/memory` | 记忆状态 / `/memory <词>` 检索 |
| `/skill` | 搜索可复用运营 Skill；`/skill 否词` |
| `/workspace` | 项目理解：`/workspace map --root .` |
| `/patch` | 结构化补丁：`/patch validate patch.json --root .` |
| `/gitops` | Git 工作流：`/gitops status --root .` |
| `/tools` `/mcp` | 列工具 / MCP 服务器 |
| `/raw` `/clear` `/exit` | 切渲染 / 清上下文 / 退出 |

对话中可直接说「**记住**…」沉淀打法、「**回忆**…」跨会话召回。
工程类问题会自动注入精简工程上下文，包括 workspace 入口/测试/配置、git 状态和 skill 状态；运营类问题仍优先注入 Amazon skills 与知识库。

长代码任务或资料整理任务默认不会自动压缩上下文，避免做到一半被摘要打断。上下文过长时会提示你手动 `/compact`；如果你确实想自动压缩，可执行 `/compact auto on`。单轮工具调用上限默认 48 步，达到上限时可直接说「继续」，或用 `ivyea config set chat_max_tool_steps 80` 调高。工程类对话会显示 `Code 计划 → 读上下文 → 修改/生成补丁 → 测试 → 复查`，工具调用也会带进度编号，便于看清它正在做哪一步。

### 1.1) 代码 Agent 闭环

Ivyea Agent 的通用代码能力不是直接“猜着改”，而是固定走一条可审计链路：项目索引 -> 任务规划 -> 代码上下文 -> 结构化 patch -> 测试 -> 失败修复计划 -> 审查门禁。

```bash
ivyea workspace index --root .
ivyea code plan "修复登录超时重试逻辑" --root .
ivyea code context "修复登录超时重试逻辑" --root .
ivyea code brief "修复登录超时重试逻辑" --root . --budget 6000
ivyea code quality --root .
ivyea code diff-brief --root .
ivyea code release-check --root . --version v0.5.7
ivyea workspace symbols RetryPolicy --root .
ivyea code impact RetryPolicy --root .
ivyea code refs RetryPolicy --root .
ivyea code rename-plan RetryPolicy --new-name RetryConfig --root .
ivyea code patch "修复登录超时重试逻辑" --root . --llm
ivyea code run "修复登录超时重试逻辑" --root . --llm-patch --max-rounds 3
ivyea code runs
ivyea code sandbox --root .
ivyea patch validate patch.json --root .
ivyea patch apply patch.json --root . --execute
ivyea code test --root . --command "python -m pytest"
ivyea code repair --root . --output-file pytest.out
ivyea code review --root .
```

关键点：

- `workspace index/search/explain` 会记录 Python 的类、函数、方法、行号、导入和调用；JavaScript/TypeScript 会提取类、函数、导出常量和调用名，便于快速定位影响面。
- `workspace symbols` 用于查符号定义；`code impact` 用于查定义、导入方、调用方和受影响测试。
- `code refs` 查定义、导入、调用和文本引用；`code rename-plan` 生成重命名 patch 草案，仍需人工 validate/apply。
- `code brief` 按字符预算输出分层上下文，适合喂给模型或人工快速审查。
- `code quality` 做本地确定性质量扫描：大文件、长函数/方法、高连接度模块、测试覆盖风险。
- `code diff-brief` 根据 git diff 生成变更摘要、建议测试和 PR 草稿。
- `code release-check` 汇总发布前只读门禁：git 状态、review gate、quality、release-plan 和建议测试。
- `code plan` 输出相关文件、步骤、风险、建议测试，并读取 `AGENTS.md`、`CLAUDE.md`、`README.md`、`pyproject.toml`、`package.json` 中的仓库约定。
- `code context` 把相关文件和仓库约定压成适合模型/人工审查的上下文。
- `code test` 会运行测试并解析 pytest 失败摘要。
- `code repair` 会把失败用例映射回测试文件和疑似生产代码，指导下一轮最小 patch。
- `code patch --llm` 默认只生成模型请求包；加 `--call` 才调用当前模型，调用后也只做 dry-run validate，不会 apply。
- `code run --llm-patch --max-rounds N` 会保存一次任务记录，并输出多轮 dry-run 状态；后续轮次等待人工 apply/test/repair 输入。
- `code runs` / `code show <id>` 用于查看已保存的 code run。
- `code sandbox` 只输出 worktree/临时目录沙箱计划，不会创建目录。
- `code review` 汇总 git diff、确定性风险扫描和建议测试。
- 这些命令不会提交、推送或发版；写文件只通过 `patch apply --execute` 明确发生。

## 四、安全护栏（动钱的底线）

- **写操作永远人工审批**：多档选项 `[1]是 [2]本会话都允许此类 [3]否 [4]改一下 [5]全部停`。
- **领星写总闸 operate**：`ivyea lingxing operate on|off|status`，**默认关**（关时只 dry-run 预览），开了也带 120 分钟自动过期。
- **硬护栏**（代码强制，LLM 绕不过）：否词 ≥15 点击 0 单才否；单次调 bid ≤±20%、步长 ≤15%；冷却 7 天；不否品牌词/竞品词/核心品类词/保护词；不投 SBV / 不操控评论；写失败自动**熔断**关开关。
- **记忆护栏**：你否过的词，下次巡检自动拦截、不再反复建议。
- **成本护栏**：`ivyea config set daily_cost_limit_cny 20` 设每日上限，超了暂停问你。
- **回滚**：每次写入留审计快照，`ivyea audit rollback <ID>` 一键回滚（否词→归档、调 bid/预算→还原旧值）。
- **计划模式 / 影子模式**：见下。

## 五、影子模式（重点）——动钱前先用数据换信任

> **这是 Ivyea Agent 独有、Claude Code/Codex/Hermes 都没有的能力。**
> 一个会动你广告预算的 Agent，最大的门槛是「我凭什么信它」。影子模式解决这个：
> **它先只看不写、把每条建议记下来，过几天用真实花费回测「如果当时照做，能省多少 / 多抓多少单」**——用你自己账户的真实数据证明它值得信，你看准了再放它真动手。

### 5.1 它怎么工作

1. **记账**：每次 `ivyea patrol --from-lingxing` 巡检，都会把候选建议（否词/收割/降bid/加bid/加预算）连同**当时的触发指标**记进影子台账（`~/.ivyea/shadow.db`）。同一店铺+杠杆+词 7 天内不重复记。
2. **只看不写**：开了影子模式后，即使你 `--execute`、即使 operate 开关是开的，**也绝不写广告**——只记录、不执行。这是比 operate 更强的一层「绝不动手」保险。
3. **回测**：过几天再 `ivyea shadow report`，它会**拉这几天的真实搜索词数据**，和当初记的建议对账：
   - **否词**：当初建议否的词，之后**还烧了多少钱、且仍 0 单** → 这就是「如果当时照做能省下的钱」。
   - **收割**：当初建议升精准的词，之后**又出了多少单** → 这就是「如果当时照做能抓住的增量」。
   - **诚实**：当初建议否、但后来**真转化出单**的词，**不计入节省**（不夸大、不给自己邀功）。

### 5.2 命令

```bash
ivyea shadow on                       # 开影子模式：巡检只记不写
ivyea shadow off                      # 关：恢复正常审批写入（仍需 operate 开关）
ivyea shadow list [--sid 1863]        # 看台账里记了哪些建议
ivyea shadow report --sid 1863 [--days 14]   # 拉近 N 天现况回测，出信任报告
```

### 5.3 推荐工作流（新店/新用户）

```
第 1 步  ivyea shadow on                              # 进影子模式
第 2 步  ivyea patrol --from-lingxing --sid 1863      # 跑巡检(只记不写)，隔几天跑一次，攒台账
   …    （这期间它一个广告都不会动）
第 3 步  ivyea shadow report --sid 1863               # 过 1~2 周，看"若照做的收益"
第 4 步  信任建立后：
        ivyea shadow off                              # 关影子模式
        ivyea lingxing operate on                     # 开领星写总闸
        ivyea patrol --from-lingxing --sid 1863 --execute   # 逐条人工审批，真动手
```

### 5.4 信任报告长什么样

```
# 影子模式信任报告 — sid 1876
> 回测了 4 条有后续数据的建议。
- 否词：2 个词若当时照做，至今已省 ¥32.50（之后仍纯烧钱0单）。
- 收割：1 个词若当时升精准，至今多抓 9 单。
  · 否词「karaoke remote」之后又烧 ¥21.50(0单) → 若照做已省
  · 否词「cheap mic」之后又烧 ¥11.00(0单) → 若照做已省
  · 收割「studio mic pro」之后又出 9 单 → 若照做已抓住
```
> 注意：上例里第 3 个被建议否的词后来转化出单了，报告**没有**把它算进节省——这种「不替自己邀功」的诚实，正是信任的来源。

### 5.5 影子模式 vs 计划模式 vs operate 开关（别混）

| 机制 | 作用范围 | 行为 |
|---|---|---|
| **计划模式** `/plan` | 单次对话 | 当轮只读、不调写工具；`/approve` 后才执行。临时「先别动手，给我看计划」 |
| **影子模式** `shadow on` | 持续，跨会话 | 巡检只记不写 + 回测换信任。长期的「先观察攒数据」 |
| **operate 开关** `lingxing operate on` | 领星写入总闸 | 关=dry-run，开=允许真写（仍需逐条审批）。影子模式开时，operate 即使开也不写 |

**优先级**：影子模式开 > operate 开关。只要影子模式开着，就绝不写。

## 六、记忆（越用越懂你的店）

- **Skills**：内置搜索词优化、否词风控、预算节奏、Listing 转化诊断、新品启动、周度复盘。对话时按问题自动注入，也可 `/skill <词>` 手动查。
- **知识库审计**：`knowledge audit` 显示来源类型、可信度、更新时间。官方知识和社区经验会分层展示。
- **用户知识库**：`knowledge import/url/rebuild/index/conflicts` 把本地 Markdown/TXT 或网页导入 `~/.ivyea/knowledge/`，来源记录在 `sources.jsonl`，含 license/body_hash；SQLite FTS5 索引支持后续扩展，冲突审计会提示用户知识与官方知识的潜在矛盾。
- **结构化画像**：除目标 ACOS/保护词外，支持毛利率、盈亏 ACOS、售价、Listing 风险，自动注入对话。
- **Listing 转化诊断**：`listing audit` 根据标题、五点、A+、搜索词、Review/评分，判断是流量问题还是承接问题，避免误否相关词。
- **Review/Offer 归因**：`review audit` 识别质量、不可用、少件、尺寸、描述不符、售后等差评/Q&A 风险，并结合评分、评论数、价格判断是否应先控广告。
- **库存/利润联动**：`offer audit` 根据毛利率、目标 ACOS、库存天数、售价/竞品价、coupon、广告 ACOS 判断能不能放量。
- **竞品/类目分析**：`competitor audit` 识别竞品流量、ASIN 串号词、类目扩展候选、保护词命中和缺失核心词。
- **周期复盘**：`weekly review` 聚合动作队列、近期巡检、Scorecard、trace、影子台账，生成周报/复盘 Markdown。
- **自动预警**：`alert check` 检查动作队列、工具失败、影子台账、画像、知识库和磁盘空间；`--notify` 可发送到 stdout、通用 webhook 或飞书机器人。
- **本地计划任务**：`schedule set/list/run-due/run` 不常驻，可由 cron/systemd/IvyeaOps 定时调用；alert 任务支持保存通知参数。
- **安全策略**：`policy init/show/check-path/check-command/explain-command` 管理 `~/.ivyea/policy.json`，可限制读写目录、命令 allow/deny，并对命令给出 allowed/risk/reasons。
- **Workspace 项目理解**：`workspace index/search/map/graph/inspect/explain` 离线扫描任意代码项目，跳过 `.git/node_modules/dist/build/venv` 等噪声，索引保存到 `~/.ivyea/workspaces/`，输出语言分布、重要文件、目录结构、符号摘要、搜索命中、依赖图、入口点、测试文件、配置文件和建议命令。`.github/workflows` 默认纳入索引，用于 CI/发版判断。
- **通用长任务**：`task create/list/show/start/step/status/log/resume` 把复杂任务保存到 `~/.ivyea/tasks/`，记录步骤状态、阻塞原因、日志和恢复提示；它是 Agent 执行长任务的进度台账，不是定时器。
- **Git/CI 只读工作流**：`gitops status/diff/workflows/release-plan` 查看仓库状态、diff 摘要、GitHub Actions workflow 和发版前检查；当前只读，真实 commit/push/release 后续接审批。
- **代码审查**：`codereview` 扫描 git diff 和未跟踪新文件，识别疑似密钥、危险 shell、宽异常捕获、生产代码无测试改动等确定性风险。
- **用户 Skill**：`skill create/audit/status/export-lock` 支持在 `~/.ivyea/skills/` 下创建个人 skill，审计触发词、正文和关联知识卡，查看用户覆盖/版本差异，并导出当前生效 skill lockfile。
- **生命周期管理**：`self status/backup/upgrade/uninstall` 识别安装方式、备份用户数据、输出升级/卸载计划；真实执行需要 `--execute` 并经过审批。
- **通用视觉**：`vision` 面向截图、UI、报表等通用视觉任务；`image vision` 继续作为 Listing 图片专用入口。默认只生成请求包，显式 `--call` 才会调用外部多模态模型。
- **图片视觉诊断**：`image audit` 先做本地资产预检查，并可导出多模态大模型审核 prompt；`image ocr` 可选调用本机 tesseract 识别图片文字；`image vision` 默认生成 OpenAI/Claude/Gemini 请求包，显式 `--call` 时才真实调用外部多模态 API。
- **运行时间线**：`trace` 记录工具调用、耗时、失败，Scorecard 会汇总。
- **业务 Eval**：`ivyea eval` 固化广告决策 golden、知识召回、skill 召回和安全脱敏，避免迭代后质量退化。

## 八、模型与健壮性

- **多模型**：`ivyea model providers` 查看完整 provider 矩阵；可用主脑包括 DeepSeek、Claude API、Gemini API、Gemini Code Assist OAuth 登录/token/probe 诊断、AWS Bedrock Converse、OpenAI 兼容、GitHub Copilot、通义、Qwen OAuth device-code/Qwen CLI 登录导入自动刷新、Kimi、Z.AI/GLM、豆包、MiniMax、OpenRouter、Nous(API key)、xAI、Ollama/本地和自定义网关。
- **降级链**：`ivyea config set fallback_models deepseek-reasoner,claude-sonnet` —— 主脑限流/挂了自动切有 key 的备用模型继续。
- **重试/退避**：429/5xx/网络错误自动重试 + 指数退避；4xx 直接报错。
- **可中断**：对话里 Ctrl+C 停本轮、不杀会话。
