# Amazon finance and settlement reconciliation

Source basis: Amazon SP-API Finances API and Settlement Reports documentation. The seller's authorized transaction data is authoritative for account-specific amounts.

## Official product boundary

- The Finances API exposes seller financial transactions and can be used to determine which released transactions make up a payment.
- Settlement reports are generated by Amazon and discovered through the Reports API; request and scheduling rules depend on the report type.
- API versions, event models, identifiers, release status, and report deprecation status must be recorded before building a reconciliation.

## Reconciliation contract

Reconcile at the smallest available transaction grain and retain:

- seller account, marketplace, currency, order and shipment identifiers;
- transaction/event type, transaction identifier, posted and released timestamps;
- principal, shipping, promotions, taxes, marketplace-withheld tax, selling/FBA fees, refunds, reserves, adjustments, and other components;
- payment or settlement identifier, bank transfer amount, and statement/report generation timestamp.

For every unexplained difference, classify it as timing, aggregation, currency, reserve/hold, refund/chargeback, fee/tax, missing data, or true mismatch.

## Guardrails

- Do not treat an estimate, order total, settlement total, and bank deposit as equivalent measures.
- Do not net currencies or marketplaces without an explicit exchange-rate and accounting policy.
- An unreleased or deferred transaction is not necessarily an error.
- Reconciliation supports investigation; it does not replace accounting records or professional accounting advice.
